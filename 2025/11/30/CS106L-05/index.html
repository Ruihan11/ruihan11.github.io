第13讲：特殊成员函数（Special Member Functions）

C++ 对象生命周期由六类特殊成员函数管理，它们是 RAII 的基石。

六大 SMF 概览

SMF 默认生成时机 作用 显式禁用
默认构造 T() 无用户定义构造时 构造对象 = delete
析构 ~T() 总生成 清理资源 = delete
拷贝构造 T(const T&) 需要拷贝时 按成员逐字段拷贝 = delete
拷贝赋值 operator=(const T&) 需要赋值时 按成员逐字段赋值 = delete
移动构造 T(T&&) C++11+，需要移动时 接管右值资源 = delete
移动赋值 operator=(T&&) C++11+，需要移动时 接管并清理右值资源 = delete

关键事实：

  • 编译器仅在首次使用时生成（且仅当无任何用户定义 SMF 时才生成所有默认 SMF）
  • 默认拷贝/移动都是 member-wise（成员依次调用其对应 SMF）——对指针是浅拷贝

为何要重写 SMFs？——指针的灾难

解决方案：深拷贝

Rule of Three（三法则）：若需自定义析构/拷贝构造/拷贝赋值其一，则三者都应自定义。因手动管理资源时，编译器生成的 SMF 必然错误。

第14讲：移动语义（Move Semantics）

问题：拷贝昂贵，临时对象可”偷”

核心思想：移动 = 接管资源 + 留下空壳

操作 拷贝构造 移动构造
时间复杂度 O(n) O(1)
内存开销 额外分配 n 字节 0
源对象状态 不变 valid but unspecified

noexcept 至关重要：STL 容器仅当移动构造 noexcept 时才启用移动，否则退化为拷贝以防异常安全问题。

lvalue vs rvalue

类型 定义 示例
lvalue 有持久身份的对象（具名、可取地址） a, *p, vec[0]
rvalue 临时对象（无名、将立即销毁） 42, takePhoto(), a + b

关键规则：

  • 拷贝用于 lvalue（持久对象，需保留原值）
  • 移动用于 rvalue（临时对象，可”牺牲”）
  • 编译器自动选择 & / && 重载

std::move：强制”移动许可”

注意：

  • std::move 本质是 static_cast<T&&>，不做移动，仅类型转换
  • 移动后对象处于 valid but unspecified 状态，禁止直接使用
  • 移动后仅允许：析构、赋值

Rule of Zero / Three / Five

法则 适用场景 实践建议
Rule of Zero 类不直接管理资源 永不写 SMF，依赖成员的 SMF
Rule of Three 类手动管理资源 必须自定义：析构 + 拷贝构造 + 拷贝赋值
Rule of Five Rule of Three + 想支持移动 五者都自定义

移动语义陷阱与最佳实践

常见错误

错误 后果 修复
移动后使用源对象 未定义行为 移动后仅允许析构/赋值
移动构造未置空源指针 double free 务必将源指针置为 nullptr
移动函数未标 noexcept STL 退化为拷贝 移动操作必须 noexcept

最佳实践

  1. 优先遵循 Rule of Zero：用 std::unique_ptr, std::vector 等 RAII 容器管理资源

  2. 移动操作必须 noexcept

  3. 移动后对象应处于”有效但未指定”状态：最小要求是可安全析构、可赋值

总结

概念 核心思想 工程价值
RAII 资源 = 对象生命周期 自动化资源管理，杜绝泄漏
Rule of Zero 委托资源管理给成员 最小化代码，最大化安全性
