TinyInfra：从零构建快 2 倍的 Llama3-8B 推理引擎

引言

大语言模型（LLM）推理优化是一个复杂但充满回报的挑战。本文将带你从零开始，逐步构建一个高性能的 Llama3-8B 推理引擎，并在 H100 GPU 上实现相比 HuggingFace baseline 2.1 倍的性能提升（从 458.80 tokens/sec 到 961.48 tokens/sec）。

我们将深入探讨以下优化技术：

  • 算法层面：KV Cache、FlashAttention、GQA
  • 计算层面：融合算子（Fused Operations）
  • 硬件层面：TF32、cuDNN 优化
  • 运行时层面：torch.compile

💡 项目地址：完整代码见 GitHub - tiny-infra

3. 融合操作（Fused Operations）：减少内核启动开销

每个 GPU 操作都涉及显著的开销：

  1. 内核启动开销：~5-10μs 每次
  2. 全局内存读取：慢
  3. 计算：快
  4. 全局内存写入：慢

对于小型操作，开销常常占主导地位。融合将多个操作合并为一个内核，消除中间的内存读写。

5. GQA（Grouped Query Attention）：高效的注意力机制

Llama-3 使用**分组查询注意力（GQA）**来减少 KV 头的数量。

GQA 原理

在标准的多头注意力中：

  • Q 头数 = K 头数 = V 头数 = 32

在 GQA 中：

  • Q 头数：32
  • KV 头数：8
  • 比例：4 个 Q 头共享每个 KV 头

这样可以显著减少 KV cache 的内存占用！

8. torch.compile：最后的 40% 加速

在所有手动优化之后，Python 开销成为瓶颈。每次解码迭代：

  1. Python 解释器执行循环
  2. 分发到 PyTorch 操作
  3. PyTorch 分发到 CUDA 内核
  4. 同步并返回 Python

torch.compile 消除了这些开销！

编译策略

编译模式对比

模式 描述 编译时间 运行速度 适用场景
default 平衡优化 快 好 通用
reduce-overhead 最小化 Python 开销 中 最快 固定形状推理
max-autotune 尝试所有实现 慢 最快 离线优化

为什么选择 “reduce-overhead”？

解码阶段以相似的形状运行相同的操作数千次。reduce-overhead 模式：

  1. 捕获 CUDA 图：记录一次内核序列，瞬间重放
  2. 消除 Python：热路径中没有解释器
  3. 减少启动开销：单个图启动 vs 数百个内核
